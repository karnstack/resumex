# Standalone Resumes - Design

**Status:** Approved
**Date:** 2026-05-02

## Goal

Make resumes render the same way template previews render. Drop the in-browser editor entirely. Drop markdown as a content format. Each resume becomes a self-contained React component + CSS, identical in shape to a template.

The user's intent: editing happens through Claude Code or a code editor - never in the browser. The browser is a renderer.

## Non-goals

- No backward compatibility with the markdown-based resumes. The single existing markdown resume (`resumes/staff-eng.md`) is replaced by the new folder form; old files are deleted.
- No editor UI of any kind. No save endpoint. No list endpoint.
- Not solving "swap a resume's look in one click." Swapping look is a manual re-port of content into a different template fork.

## Architecture

### File layout

Templates and resumes share one shape:

```
templates/<id>/
  index.tsx     # default export: standalone React component, no props
  styles.css    # scoped styles
  meta.ts       # { id, name, description }

resumes/<variant>/
  index.tsx     # default export: standalone React component, no props
  styles.css    # scoped styles
  meta.ts       # { variant, displayName }
```

Templates contain placeholder content (the existing sample data, baked into JSX). Resumes contain the user's real content.

### Component contract

```tsx
import './styles.css'
import { PageFrame } from '@/components/page-frame/PageFrame'

export default function MyResume() {
  return (
    <PageFrame fitDeps={[]} className="..." innerClassName="...">
      {/* JSX with content baked in */}
    </PageFrame>
  )
}
```

No props. No `ParsedResume` type. The component is a fully self-contained page.

`PageFrame` is required so fit-to-page measurement and print CSS work uniformly across templates and resumes.

### Routes

| Route | Purpose | Notes |
| --- | --- | --- |
| `/` | Home: list resumes + show templates | Copy updated to remove "edit in the ui" wording. Listing pulls from generated registry. |
| `/$variant` | Render a resume | Mirrors `/preview/$templateId` shell: `Stage` + `PageFrame` + `PreviewToolbar`. Print button = `window.print()`. |
| `/preview/$templateId` | Render a template | Simplified: renders the template component directly. No more `loadSample()`. |
| `/templates` | Template gallery | Unchanged. |
| ~~`/$variant/print`~~ | (deleted) | Print works in-place via the toolbar. |

### Registries

- `src/generated/templates.ts` - already auto-generated by `sync-templates-plugin`. Continues to work; type changes from "template that takes ParsedResume" to "standalone component".
- `src/generated/resumes.ts` - new, auto-generated by a parallel `sync-resumes-plugin`. Scans `resumes/<variant>/` directories, emits `import` statements + a registry object.

Both plugins re-run on file add/remove during dev. Restart-on-change is acceptable; HMR for content edits inside `index.tsx` works through Vite directly.

### Mode gating

`isPublic` (`VITE_RESUMEX_MODE=public`) currently gates editor surfaces. With editor gone:

- `/` - keep redirect to GitHub in public mode (don't expose your fork's source).
- `/$variant` - **relax** the public-mode block. Resumes are now bundled TS, so they ship with the deploy. Public visitors can view them.
- `/templates`, `/preview/$templateId` - relax too (templates were always meant to be browseable).

Net effect: in public mode, only the home page redirects. Everything else renders.

## Skill rewrite: `writing-resumes`

The skill operations change shape because the source artifact is now JSX, not markdown.

### Create new resume

User says "create a new resume from my linkedin" or similar.

1. Ask three things in one message:
   - Source material (PDF path, pasted text, or describe-from-scratch).
   - Variant name (kebab-case; default `default`).
   - Template to fork from (list `templates/<id>` excluding `_starter`; default `emerald-twocol`).
2. Copy `templates/<id>/*` to `resumes/<variant>/` via Bash `cp -r`.
3. Edit `resumes/<variant>/meta.ts` to set the variant id and display name.
4. Edit `resumes/<variant>/index.tsx` to replace placeholder content with the user's real data. Use `Edit` for targeted replacements; `Write` only if rewriting wholesale. Be conservative: don't fabricate.
5. Confirm and suggest opening `http://localhost:5173/<variant>`.

### Iterate

User says "tighten the karnstack bullet" or similar.

1. Read `resumes/<variant>/index.tsx`.
2. Apply changes with `Edit` (preferred) or `Write` (full rewrite).
3. Briefly state what changed.

### Fork

User says "fork frontend as senior-frontend".

1. Verify `resumes/<to>/` doesn't exist.
2. `cp -r resumes/<from>/ resumes/<to>/` via Bash.
3. Update `resumes/<to>/meta.ts` to the new variant id.
4. Suggest immediate iteration.

### Swap look

User says "swap the look of staff-eng to minimal-mono".

This is a manual re-port. Steps:

1. Capture the current resume's content (read `resumes/<variant>/index.tsx`).
2. Copy `templates/<new-id>/index.tsx` and `styles.css` over the resume's.
3. Re-insert the user's content into the new layout, mapping section-by-section. Be careful with structural differences between templates.
4. Acknowledge to the user that this is a rewrite, not a one-liner.

### Delete / rename

`rm` and `mv` via Bash. Skill doesn't manage these specially.

### What the skill no longer talks about

- YAML frontmatter
- Markdown conventions (`### Title @ Subtitle`, italic metadata, bullet patterns)
- `ParsedResume` schema
- The parser

## Refactor work

### Templates

Each `templates/<id>/index.tsx` currently signs as `(props: { resume: ParsedResume }) => JSX`. Refactor to a no-prop component, with the existing sample data inlined as JSX. The two existing templates:

- `templates/emerald-twocol/`
- `templates/minimal-mono/`
- `templates/_starter/` (starter; refactor too)

The current sample data lives in `.gallery/sample.md`. The refactor inlines that content (parsed structure, written as JSX) into each template. After this, `.gallery/sample.md` is deleted.

### Routes

- `routes/$variant.tsx`: rewrite as a copy of `routes/preview.$templateId.tsx`, but loading from the resumes registry instead of the templates registry. Drop `Editor` import and the editor-mounting logic.
- `routes/preview.$templateId.tsx`: drop `loadSample()` call; render the template component directly (it now contains its own content).
- `routes/index.tsx`: replace the markdown-listing fetch with a sync read from the generated resumes registry. Update copy:
  - Remove "talk to claude code or edit in the ui."
  - Replace `.md` badge with whatever fits (e.g. `.tsx`, or just drop the badge).

### Registry types

`src/lib/template-registry.ts` types `ResumeTemplate` as a component receiving `{ resume: ParsedResume }`. Change to `() => JSX.Element`. Same for the new resume registry.

## Print

`PreviewToolbar` already has a working print button that calls `window.print()`. The print CSS hides the toolbar (`data-print-hide`) and the stage background. `PageFrame` has print-specific layout.

`/$variant` reuses this exact toolbar. Nothing new to build for print.

## Conventions docs

- `docs/CONVENTIONS.md` - rewrite around standalone components. Drop frontmatter/markdown sections. Keep project-level conventions (mode gating, filesystem-as-source-of-truth, no DB, no network).
- `docs/TEMPLATE_GUIDE.md` - rewrite around the new component contract. Same shape applies to resumes too; note this explicitly.

Keep them as separate files: `CONVENTIONS.md` for project-level rules, `TEMPLATE_GUIDE.md` for the component contract.

## Validation / verification

- Dev server boots cleanly.
- `pnpm typecheck` passes (after schema and parser are gone, types should still resolve everywhere).
- `pnpm test` - some tests will be deleted (parser tests). What remains should pass.
- Manual: open `/`, `/$variant`, `/preview/$templateId`, `/templates`. All four render. Print button on `/$variant` produces a clean PDF.
- The existing `staff-eng` resume (re-created as `resumes/staff-eng/`) renders the same content it did before.

## Out of scope

- Per-resume `meta.ts` schema beyond `{ variant, displayName }`. Anything richer (e.g. tags, target role) can be added later.
- A "list of all my resumes" page beyond the home grid.
- Public-mode polish.
